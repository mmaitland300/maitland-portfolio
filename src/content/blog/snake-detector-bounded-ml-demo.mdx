## The Problem With Old Notebook Projects

Notebook projects are great for getting an idea moving. They are not always great at showing what still works six months later.

Snake Detector started in that familiar shape: a computer-vision experiment that could train and predict, but lived too close to the original exploratory workflow. The interesting part was not just whether a model could classify `snake` or `no_snake`. It was whether the project could become something another person could inspect without needing my local notebook, my local folders, or my memory of how the run was assembled.

The subject was not random. I kept snakes and other reptiles when I was younger, so the image domain was already interesting to me. That made it a good small ML project: personal enough to keep my attention, narrow enough to keep the scope honest.

The weak parts were outside the prediction call:

- scripts and notebooks were too easy to run in only one environment
- dataset status was mixed together with model behavior
- the live demo path was separate from the repo story
- release artifacts were not clearly named or versioned
- the README could not quickly explain what was real, what was placeholder-safe, and what was out of scope

The cleanup work was not about making the project sound bigger. It was about turning an old Colab-style demo into a repo-shaped project with enough structure to publish.

## What Changed In The Refactor

The first pass was basic engineering hygiene:

- move the maintained workflow into a package and CLI
- keep old Colab-era scripts under `legacy/`
- document `split -> train -> eval -> predict` instead of relying on notebook cell order
- add CI so tests and linting run outside my machine
- separate the local placeholder-safe baseline from the real model powering the hosted demo

That last point mattered most. The live demo now uses a real-photo iNaturalist-trained Keras model, but the repo still needs a reproducible path that does not require committing raw third-party wildlife images. Those are different jobs. Treating them as different lanes made the README and release story much easier to explain.

## The Boundary Came First

The most useful decision was to narrow the claim before improving the packaging.

Snake Detector is not a field safety tool. It is not species identification. It does not tell someone whether an animal is dangerous. It is a bounded image-classification demo with one job: separate whole images into `snake` and `no_snake` classes under documented conditions.

That boundary changes the whole project. Instead of stretching the README toward impressive language, the repo can explain exactly what exists:

- a live Hugging Face demo backed by a real-photo iNaturalist-trained Keras model
- a GitHub Release for the model, metrics, threshold sweep, config, and evaluation assets
- a package-based CLI for split/train/eval/predict workflows
- model and dataset cards that explain scope, data posture, metrics, and limits
- a placeholder-safe local benchmark so the engineering flow can be reproduced without committing third-party wildlife photos

The narrower claim is actually stronger because it gives reviewers something concrete to inspect.

## Why The Placeholder Baseline Still Exists

The placeholder-safe benchmark is not trying to prove the model is useful in the real world. It proves the project shape still runs.

That distinction can feel awkward at first. The live model is the stronger public demo, but the generated placeholder data is the safer reproducibility path. I kept both because they answer different reviewer questions:

- "Can I inspect a real trained model and metrics?" Yes, through the release package.
- "Can I run the project workflow without downloading restricted images?" Yes, through the placeholder-safe baseline.
- "Are these the same thing?" No, and the repo should say that plainly.

That is the kind of boring clarity that makes an ML repo easier to trust.

## Separating The Model From The Package

One small but important cleanup was versioning.

The Python package still has its own version because the CLI/API surface is one thing. The trained model release is another thing. Treating both as the same version would make the repo feel cleaner for about ten seconds, then confusing forever.

So the package can stay at `0.1.0` while the model release is named `v1.1.0-real-dataset`. That release label says what changed: the public demo is no longer only a placeholder-safe engineering baseline; it is backed by a real-photo dataset lane and published model assets.

That distinction matters because future changes might update the app without changing the model, or update the model without changing the Python package. Keeping those lanes separate makes the project easier to reason about.

## Data Rights Shape The Repo

The uncomfortable part of public ML work is that "I trained a model" is not the same as "I can publish everything I trained on."

For Snake Detector, raw third-party images stay out of git. The repo documents dataset lanes instead:

- the older local prototype corpus, which is excluded until rights are verified
- the generated placeholder dataset, which is safe for reproducible local checks
- the real-image retraining lane, which is documented through source/license notes and release artifacts rather than raw image redistribution

That is less flashy than dropping a folder full of images into the repo, but it is much easier to stand behind.

## What The Release Adds

The real model release gives the project a better center of gravity.

The live demo uses a Keras model trained on iNaturalist-sourced images. The release package mirrors the model artifact, configuration, held-out metrics, threshold sweep, and visual evaluation assets. The README can now point to the release instead of asking readers to infer what the demo is running.

That also makes the placeholder benchmark less confusing. It is still useful, but for a different reason. It proves the local engineering path can run without distributing restricted images. It is not the thing powering the live demo.

## What I Still Won't Claim

The more public evidence a project has, the more tempting it is to make the language louder. I am trying to do the opposite.

I still would not claim:

- species identification
- field-ready wildlife safety
- broad real-world robustness
- complete dataset redistribution
- that a single confidence score should drive a safety decision

Those limits are not disclaimers pasted on at the end. They define the project.

## What I Would Reuse

The main pattern I would reuse is this:

1. Move the maintained path out of notebook memory and into package commands.
2. Decide the smallest true claim.
3. Separate data, model, package, and demo versions.
4. Publish enough artifacts to inspect the result.
5. Keep restricted data out of the repo.
6. Write the README so the limits are visible before the charts.

That pattern is useful beyond computer vision. Research prototypes, audio tools, and support utilities all get easier to evaluate when the repo says what is real, what is provisional, and what is intentionally out of scope.

For Snake Detector, the outcome is not a grand ML product. It is something better for this stage: a small demo with a real model, a reproducible project shape, and boundaries clear enough that I do not have to oversell it.
